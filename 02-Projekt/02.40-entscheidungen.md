# Entscheidungen – EMC Mitgliederverwaltung

# 1. Zweck des Dokumentes

Dieses Dokument dient als langfristiges Projektgedächtnis der EMC Mitgliederverwaltung.

Es dokumentiert strategische, organisatorische, fachliche und technische Entscheidungen, die Auswirkungen auf Produkt, Projekt oder Dokumentation besitzen.

Ziel ist die langfristige Nachvollziehbarkeit wesentlicher Entscheidungen.

---

# 2. Governance für Entscheidungen

## Nummerierung

Jede Entscheidung erhält eine eindeutige Kennung.

Schema:

```text

E-003
...
```

---

| Status    | Bedeutung                                                 |
| --------- | --------------------------------------------------------- |

# E-017

## Titel

Spring Boot Actuator ist Standard für Backend-Monitoring

**Status:** Aktiv

**Datum:** 12.06.2026

### Kontext

Mit Einführung von BL-007 Backend Healthchecks wurde das bisherige Monitoring um anwendungsbezogene Healthchecks erweitert.

Reine Container- oder Portprüfungen liefern keine ausreichende Aussage über die tatsächliche Betriebsbereitschaft der Anwendung.

### Entscheidung

Spring-Boot-Backends werden standardmäßig über Spring Boot Actuator überwacht.

Mindestens bereitzustellen sind:

```text
/actuator/health
/actuator/health/readiness
```

### Begründung

Actuator-Healthchecks ermöglichen die Bewertung der tatsächlichen Betriebsbereitschaft der Anwendung.

Insbesondere berücksichtigt der Readiness-Check zusätzlich die Datenbankerreichbarkeit und besitzt damit einen höheren Aussagewert als reine Container- oder Netzwerkprüfungen.

### Auswirkungen

- neue Spring-Boot-Backends müssen Actuator bereitstellen
- Monitoring erfolgt bevorzugt über Health- und Readiness-Endpunkte
- Uptime Kuma nutzt Actuator-Endpunkte als primäre Betriebsprüfung
- Datenbankverfügbarkeit wird Bestandteil des Betriebsmonitorings

# E-018

## Titel

Laufzeitumgebungen werden über Spring Profiles unterschieden

**Status:** Aktiv

**Datum:** 12.06.2026

### Kontext

Mit BL-007 wurde die technische Umgebungserkennung eingeführt.

Für Deployment, Monitoring und Fehleranalyse muss die aktive Zielumgebung eindeutig identifizierbar sein.

### Entscheidung

Die Laufzeitumgebung wird über das Spring-Profil gesteuert.

Grundlage:

```text
SPRING_PROFILES_ACTIVE
```

Zuordnung:

```text
leer   -> LOCAL
dev    -> DEV
prod   -> PROD
```

### Begründung

Die Verwendung von Spring Profiles entspricht dem etablierten Spring-Boot-Standard und ermöglicht eine eindeutige Identifikation der aktiven Umgebung.

Dadurch können Deployments, Monitoring und Fehleranalysen zuverlässiger durchgeführt werden.

### Auswirkungen

- DEV- und PROD-Deployments setzen `SPRING_PROFILES_ACTIVE` explizit
- die aktive Umgebung wird über `/api/system/info` bereitgestellt
- Fehlkonfigurationen können unmittelbar erkannt werden
- zukünftige Frontend-Anzeigen können die Umgebungsinformation nutzen
- Deployment- und Smoke-Tests prüfen die aktive Laufzeitumgebung

# E-019

## Titel

Hybrides Reportingmodell für die EMC Mitgliederverwaltung

**Status:** Aktiv

**Datum:** 14.06.2026

### Kontext

Im Rahmen von BL-012 Reportingstrategie wurde die bestehende Access-Reporting-Landschaft analysiert und eine langfristige Zielarchitektur bewertet.

Zur Auswahl standen:

- fest programmierte Berichte
- templatebasierte Berichte
- generische Reportingplattformen
- hybride Ansätze

### Entscheidung

Die EMC Mitgliederverwaltung verfolgt künftig ein hybrides Reportingmodell.

Die fachliche Logik wird in zentralen Reporting-Services des Spring-Boot-Backends implementiert.

Die Ausgabe erfolgt formatabhängig über PDF-, Excel-, CSV- und Dokumentgeneratoren.

Eine generische Reportingplattform wird zunächst nicht eingeführt.

### Begründung

Die bestehende Reporting-Landschaft ist fachlich überschaubar.

Ein hybrider Ansatz bietet ausreichende Flexibilität bei geringer technischer Komplexität und bleibt gleichzeitig kompatibel mit zukünftigen Anforderungen aus EMC Finanzen.

### Auswirkungen

- Einführung zentraler Reporting-Services
- Trennung von Fachlogik und Darstellung
- Unterstützung von PDF-, Excel- und CSV-Ausgaben
- spätere Erweiterbarkeit um Dashboards
- keine Einführung einer eigenständigen Reportingplattform

# E-021

## Titel

Image-basiertes Deployment für Frontend und Backend

**Status:** Aktiv

**Datum:** 15.06.2026

### Kontext

Im Rahmen von BL-008 Frontend PROD Image Deployment wurde das Frontend-Deployment an den bereits etablierten Backend-Standard angepasst.

Zuvor wurde das Frontend PROD über Bind-Mounts für dist-Verzeichnis und nginx-Konfiguration betrieben, während das Backend bereits image-basiert deployt wurde.

### Entscheidung

Frontend und Backend werden grundsätzlich image-basiert deployt.

Versionierte Docker Images stellen die verbindlichen Deployment-Artefakte dar.

DEV verwendet Snapshot-Versionen:

```text
X.Y.Z-SNAPSHOT
```

PROD verwendet Release-Versionen:

```text
X.Y.Z
```

### Begründung

Image-basierte Deployments ermöglichen:

- reproduzierbare Deployments
- eindeutige Versionszuordnung
- vereinfachte Rollback-Prozesse
- einheitliche Betriebsprozesse
- konsistente DEV-/PROD-Strategien

### Auswirkungen

- Frontend und Backend folgen demselben Deploymentmodell
- Deployment-Artefakte werden als Docker Images archiviert
- Rollbacks erfolgen über frühere Image-Versionen
- Compose-Dateien und Portainer-Stacks referenzieren versionierte Images
- Bind-Mount-basierte Frontend-Deployments werden nicht mehr verwendet

# E-022

## Titel

Compose-Konfigurationen müssen konsistent geführt werden

**Status:** Aktiv

**Datum:** 15.06.2026

### Kontext

Mit der Einführung versionierter Docker Images für Frontend und Backend wurde deutlich, dass die tatsächliche Deployment-Version von mehreren Konfigurationsebenen abhängt.

Abweichungen zwischen Repository, NAS und Portainer können dazu führen, dass trotz korrekt gebauter Images eine falsche Version deployt wird.

### Entscheidung

Die Compose-Konfigurationen in folgenden Bereichen müssen konsistent gepflegt werden:

```text
EMC-INFRA
→ compose/.../docker-compose.yml

NAS
→ /volume1/docker/compose/.../docker-compose.yml

Portainer
→ Stack Definition
```

Insbesondere müssen Image-Versionen in allen drei Bereichen identisch geführt werden.

### Begründung

Nur konsistente Compose-Konfigurationen gewährleisten reproduzierbare Deployments und nachvollziehbare Rollback-Prozesse.

### Auswirkungen

- Versionsänderungen erfordern Anpassungen in allen drei Konfigurationsebenen
- Deployment-Checklisten prüfen die Konsistenz der Compose-Dateien
- Portainer gilt nicht als alleinige Source of Truth
- EMC-INFRA bleibt die dokumentierte technische Referenz
- Redeployments dürfen erst nach erfolgreicher Konsistenzprüfung erfolgen
- Smoke Tests überprüfen die tatsächlich gestartete Version
